React + Ynex UI Implementation Plan

Complete phase-by-phase blueprint for migrating Himalayan Bullion from Razor Pages to a React SPA — backed by the existing .NET 9 API.

📅 Prepared March 2026 ⚛️ React 18 + TypeScript 🎨 Ynex Template 🔌 60+ API Endpoints 9 Phases
📋 Project Overview
🎯

What We Are Building

React SPA consuming existing .NET 9 REST API — no backend changes required

The existing Razor Pages frontend will be replaced with a modern React single-page application built on the Ynex admin dashboard template. The HimalayanBullion.Api project remains unchanged — it already exposes all required endpoints with JWT authentication and policy-based authorization. The React app will talk exclusively to the API over HTTPS.

DimensionCurrent (Razor Pages)Target (React + Ynex)
RenderingServer-side (SSR)Client-side SPA (CSR)
AuthCookie / IdentityJWT stored in httpOnly cookie or memory
APIRazor Pages own DbContext100% via /api/* endpoints
StylingBootstrap + custom CSSYnex + Tailwind CSS
StateViewModels / TempDataRedux Toolkit + React Query
RoutingRazor Page routesReact Router v6
FormsModel bindingReact Hook Form + Zod validation
TablesHand-rolled HTMLTanStack Table v8
ChartsChart.js CDNRecharts / ApexCharts
💡
Backend stays untouched. All 60+ API endpoints in HimalayanBullion.Api are consumed as-is. The only backend change needed is adding a CORS policy to allow the React dev server origin during development.

Two decoupled projects communicating over REST + JWT

🌐 Browser — React SPA (Ynex Template)  |  Port 5173 (dev) / Nginx (prod)

🔌 HimalayanBullion.Api (.NET 9 Web API)  |  Port 5001 — All existing controllers
⬇ EF Core ⬇

React App Internal Layers

🎨 Pages / Views  (Ynex layout + page components)
🧩 Feature Modules  (orders, invoices, payments, items, customers, reports, settings, portal)
🗂️ State Layer  (Redux Toolkit slices + React Query hooks)

🛡️ Auth Guard  (PrivateRoute + role/permission checks)

// src/api/axiosInstance.ts const api = axios.create({ baseURL: '/api' }); api.interceptors.request.use((config) => { const token = store.getState().auth.token; if (token) config.headers.Authorization = `Bearer ${token}`; return config; }); // 401 → attempt refresh → retry original request api.interceptors.response.use( (res) => res, async (error) => { if (error.response?.status === 401) { await store.dispatch(refreshTokenThunk()); return api(error.config); } return Promise.reject(error); } );
6
Add CORS to .NET API (one-time backend change)
In HimalayanBullion.Api/Program.cs add:
builder.Services.AddCors(o => o.AddPolicy("ReactApp", p => p .WithOrigins("http://localhost:5173", "https://yourdomain.com") .AllowAnyHeader() .AllowAnyMethod() .AllowCredentials())); app.UseCors("ReactApp"); // before UseAuthorization

ConnectIPS Payment Flow

1
User clicks "Pay via ConnectIPS" on invoice
Calls POST /api/payments/initiate → returns redirectUrl + qrCode
2
React opens ConnectIPS gateway in new tab or redirects
Use window.location.href = response.redirectUrl
3
ConnectIPS posts back to /Payments/ConnectIPSCallback
This hits the existing Razor Page callback (unchanged). After success, redirect user back to portal invoice page.
4
React portal page polls / shows updated status
GET /api/payments/order/{orderId}/status — show success toast + reload invoice

🗂️ State Management Strategy
🗂️

Redux Toolkit + React Query Division of Responsibility

WhatHowWhy
Auth token, user info, permissionsRedux Toolkit sliceNeeded by interceptor + guards — must be global, synchronous
Sidebar open/close, themeRedux Toolkit slicePersistent UI state, not tied to server
Orders list, detail, paginationReact Query useQueryAutomatic caching, background refetch, stale-while-revalidate
Create/Edit/Delete mutationsReact Query useMutationHandles loading, error, success states + cache invalidation
Form stateReact Hook FormMinimal re-renders, Zod integration
Toast notificationsreact-hot-toastFire-and-forget from mutation callbacks

Example: useOrders Hook

// src/hooks/useOrders.ts export const useOrders = (params: OrdersFilterParams) => useQuery({ queryKey: ['orders', params], queryFn: () => ordersApi.getAll(params), staleTime: 30_000, // 30 seconds fresh placeholderData: keepPreviousData, // no layout shift on page change }); export const useCreateOrder = () => useMutation({ mutationFn: ordersApi.create, onSuccess: () => { queryClient.invalidateQueries({ queryKey: ['orders'] }); toast.success('Order created successfully'); }, onError: (err) => toast.error(err.message), });

Coding Conventions for the React Project

Naming

  • → Components: PascalCase.tsx
  • → Hooks: camelCase.ts prefixed with use
  • → API files: camelCase + Api.ts (e.g. ordersApi.ts)
  • → Types: PascalCase interfaces, exported from types/*.ts
  • → Zod schemas: camelCaseSchema (e.g. createOrderSchema)

Component Pattern

  • → Every page = features/module/PageName.tsx
  • → Shared UI = components/ui/
  • → No inline API calls — always go through a hook
  • → No direct axios imports — use axiosInstance

Security

  • → Never store JWT in localStorage directly — use Redux with sessionStorage fallback or httpOnly cookie mode
  • → All admin routes behind <AuthGuard>
  • → All permission-gated actions also checked server-side (API returns 403)
  • → Sanitise any HTML rendered from API data (use DOMPurify)

Error Handling Pattern

  • → API errors: extract response.data.message from ApiResponse<T>
  • → Network errors: generic "Connection failed" toast
  • → 403: redirect to /403 page
  • → 401 after refresh failure: clear auth + redirect to /login
✅
Backend API Contract: The ApiResponse<T> wrapper is returned by every endpoint — always check response.data.success before accessing response.data.data. Create a typed Axios response interceptor that throws if success === false, so all hooks get a clean rejection to handle.
